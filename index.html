Approach

How I decide before I ship

Same clarity at 7am or midnight — the process does not change when tired. Invert the failure, find the real constraint, prove the number moved, ship only what survives contact with reality.

01

Correctness first

Before throughput or clever design — what happens on retry, duplicate delivery, or partial failure? Usage, bind, and billing-adjacent paths cannot tolerate silent drift. I default to explicit state and recoverable errors.

02

Measure the bottleneck

Focus beats motivation when the constraint is real — p99 in a transform path, deploy friction blocking three teams, an observability gap hiding retries. Name it, cut it, show the graph moved.

03

Invert the failure

Ask what breaks at 10× traffic, a bad deploy, or a confused downstream owner — then design that version of the story out of existence. Most of my best decisions began as failures I refused to allow.

04

Design for who inherits it

Platform work compounds when the next team’s safe path is the easy path — clear contracts, governed releases, honest docs in review. Steady in incident; direct about what the next increment should be. Step in when needed; step back when someone else should lead.

Safe deploy — the decision chain, not a hero deploy

Deploy window ON

I invert the failure first: what if east promotes while west still serves stale? DynamoDB deploy window gates routing accounts before any regional cut.

East → health check

CDK promotes to primary region; automated health verification must pass before west is even eligible — fail closed, not hope closed.

Approval → West

Human approval between regions with cross-region replica verification — tradeoff: slower promotion vs. silent skew members would feel.

Window OFF · rollback

Deploy window closes; failed checks trigger auto-rollback. Two product orgs inherited this as default — constraints other squads design inside.

What teams inheritShared safe-deploy stack, CDK constructs, CloudFront cache policy for deployment-phase routing, failure-mode runbooks — platform judgment encoded so the next engineer’s safe path is the easy path.
What I refused to allowFriday surprise deploys, regional skew without observability, poison events reaching CRM, bind state drifting under retry — each became a constraint in code or pipeline.

Correctness at scale

Idempotency, deduplication, observability — when state represents usage or money, ambiguity is a bug class, not an edge case.
